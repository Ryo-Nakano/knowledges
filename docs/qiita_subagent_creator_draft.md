## 🔰 はじめに

Gemini CLI には「サブエージェント」という機能があります。
特定のタスクに特化した専門エージェントをローカルに定義し、メインエージェントの「ツール」として呼び出せる、強力な拡張の仕組みです。

ただし 2026年3月現在、**サブエージェントは Experimental（実験的機能）** の段階です。
Claude Code のカスタムサブエージェントと比べると、Gemini CLI での活用事例や解説記事はまだ多くありません。

そこで私は、**「Gemini CLI 自身にサブエージェントの作成方法をレクチャーし、ユーザーの作成作業を支援してくれる Skill（`subagent-creator`）」** を作りました。

この記事では、その制作過程で得た知見をお伝えします。


## 🧞 この記事で得られる3つの学び

記事をとおして、以下の3つをお持ち帰りいただけます。

1. **「Skill」という設計パターンを理解できる**
   「やりたいことを AI に覚えさせ、任意のタイミングで呼び出す」仕組みがどう機能するかを把握できます。

2. **自律改善ループの実装方法を知ることができる**
   「使えば使うほど AI が賢くなっていく」仕組みを、プロンプトレベルでどう実現するかを学べます。

3. **「スキル化」という考え方を自分のプロジェクトに応用できる**
   `subagent-creator` に限らない、再現性のある AI 活用の設計思想を持ち帰れます。

では早速、本題へ。


## 📖 Gemini CLI のサブエージェントとは

サブエージェントは、**YAMLフロントマターを持つ Markdown ファイル** として定義された専門エージェントです。
メインの Gemini CLI エージェントが「どのサブエージェントを使うか」を自動でルーティングしてくれます。

```yaml
---
name: code-reviewer
description: コードの品質・可読性・バグの可能性をレビューする専門エージェント。コードレビューやリファクタリングの提案を求められたときに使用する。
kind: local
tools:
  - read_file
  - grep_search
---

（ここにシステムプロンプトを書く）
```

有効化には `~/.gemini/settings.json` に以下を追記します。

```json
{
  "experimental": {
    "enableAgents": true
  }
}
```

ファイルの配置場所は2種類あります。

| 対象 | パス |
|---|---|
| プロジェクト共有 | `.gemini/agents/` |
| 個人用 | `~/.gemini/agents/` |

:::note warn
⚠️ サブエージェントは **YOLOモード**（ユーザーに確認を求めずツールを実行するモード）で動作します。`write_file` や `run_shell_command` を付与する際は、システムプロンプトで操作範囲を厳格に制限してください。
:::

## 🤔 なぜスキルを作ろうと思ったのか

最初は「自分でサブエージェントの作成方法を調べて、設定ファイルを手書きしよう」と思っていました。

でも、ふと気づきます。

> 「毎回手書きするより、**Gemini CLI 自身に作成方法をレクチャーして、作成支援させた方が圧倒的に再現性が高い**のでは？」

これが `subagent-creator` 開発のスタートです。

「AI でサブエージェントを作る」= **「AI を作るための AI」** というメタな構造。
自分でやるより、仕組みを作ってしまう方が生産性が高い、という考え方の転換です。


## 🛠️ subagent-creator の概要

### ディレクトリ構成

```text
skills/subagent-creator/
├── SKILL.md                      ← メインのスキル定義
└── references/
    ├── examples.md               ← サブエージェント定義の完成例
    ├── advanced_topics.md        ← Hooks / MCP / A2A などの発展的トピック
    └── known_issues.md           ← 過去の問題と対策の記録
```

### ワークフロー（Step 1〜6）

`subagent-creator` を起動すると、以下のステップで対話的にサブエージェントを作成してくれます。

| Step | 内容 |
|---|---|
| 1 | 環境確認（`settings.json` のチェック、過去の知見の読み込み） |
| 2 | 要件のヒアリング（目的・使用ツール・配置場所） |
| 3 | 任意項目の提案（モデル・temperature・max_turns 等） |
| 4 | サブエージェント定義の生成 |
| 5 | ユーザー確認のうえファイルを作成 |
| 6 | 動作確認の案内（`/agents list` 等） |

「サブエージェントを作りたい」と伝えるだけで、対話形式で必要な情報をヒアリングし、定義ファイルを生成・配置してくれます。

### セキュリティへの配慮

サブエージェントは YOLO モードで動作するため、ツールの選択が特に重要です。`subagent-creator` は、リスクに応じた3つのレベルを定義した `examples.md` を参照します。

**低リスク例：コードレビューエージェント**

```yaml:code-reviewer.md
---
name: code-reviewer
description: コードの品質・可読性・バグの可能性をレビューする専門エージェント。コードレビューやリファクタリングの提案を求められたときに使用する。
kind: local
tools:
  - read_file
  - grep_search
  - glob
  - list_directory
model: gemini-3.1-pro-preview
temperature: 0.2
---

あなたはコードレビューの専門家です。以下の原則に従ってレビューを行ってください。

## レビュー観点

1. **可読性**: 変数名・関数名は意図を表しているか
2. **保守性**: 重複コードはないか、適切に関数化されているか
3. **バグリスク**: エッジケースは考慮されているか、エラーハンドリングは適切か
4. **パフォーマンス**: 明らかな非効率はないか

## 出力フォーマット

発見した問題ごとに以下の形式で報告すること：

- **箇所**: ファイル名と行番号
- **深刻度**: 高 / 中 / 低
- **指摘内容**: 問題の説明
- **改善案**: 具体的な修正提案

最後に、レビュー対象の全体的な品質を「良好 / 要改善 / 要修正」の3段階で総括すること。
```

**高リスク例：プロジェクト初期化エージェント**（`run_shell_command` を含む場合）

```yaml:project-initializer.md
---
name: project-initializer
description: 新しいプロジェクトの雛形を作成する専門エージェント。プロジェクトの初期化やボイラープレートの生成を求められたときに使用する。
kind: local
tools:
  - read_file
  - write_file
  - run_shell_command
  - list_directory
  - ask_user
model: gemini-3-flash-preview
temperature: 0.3
max_turns: 20
---

あなたはプロジェクト初期化の専門家です。ユーザーの要望に合わせてプロジェクトの雛形を作成します。

## 制約事項（必ず遵守）

1. **ファイル操作の制限**: ファイルの作成・書き込みは、ユーザーが指定したプロジェクトディレクトリ内のみで行うこと。
2. **シェルコマンドの制限**: 以下のコマンドのみ実行を許可する：
   - `mkdir -p`（ディレクトリ作成）
   - `npm init` / `npx`（パッケージ初期化）
   - `git init`（リポジトリ初期化）
   - `ls`, `cat`, `pwd`（状態確認）
3. **禁止操作**: `rm`, `sudo`, `curl | sh` 等の破壊的コマンドは絶対に実行しないこと。
4. **確認の徹底**: 不明な点がある場合は `ask_user` ツールでユーザーに確認すること。

## ワークフロー

1. ユーザーにプロジェクトの種類・言語・フレームワークを確認する
2. プロジェクトディレクトリを作成する
3. 必要なファイル（README.md, .gitignore 等）を生成する
4. パッケージマネージャの初期化を行う
5. 作成したファイル一覧をユーザーに報告する
```

:::note info
💡 `run_shell_command` は「許可コマンドのみホワイトリストで明記 + 禁止コマンドも列挙」するのが安全設計のポイントです。
:::

<details><summary>中リスク例：ドキュメント生成エージェント</summary>

```yaml:doc-generator.md
---
name: doc-generator
description: コードベースを解析してAPIドキュメントやREADMEを自動生成する専門エージェント。ドキュメントの作成・更新を求められたときに使用する。
kind: local
tools:
  - read_file
  - write_file
  - grep_search
  - glob
  - list_directory
  - ask_user
model: gemini-3-flash-preview
temperature: 0.3
---

あなたはテクニカルライティングの専門家です。コードベースを解析し、正確で読みやすいドキュメントを生成します。

## 制約事項（必ず遵守）

1. **書き込み先の制限**: ドキュメントファイル（`.md`, `.txt`）の作成・更新のみ行うこと。ソースコードの変更は禁止。
2. **既存ファイルの扱い**: 既存のドキュメントを上書きする前に、必ず `ask_user` で確認すること。
3. **内容の正確性**: コードから読み取れる情報のみを記述し、推測や憶測を含めないこと。

## ワークフロー

1. 対象のコードベースを `read_file` と `grep_search` で調査する
2. 公開API（関数・クラス・メソッド）を特定する
3. ドキュメントのドラフトを `ask_user` で提示し、確認を得る
4. 承認後、`write_file` でドキュメントファイルを作成する
```

</details>


## 💡 設計のハイライト：自律的な学習ループ

`subagent-creator` で最もこだわった設計が、**「使うたびに自動で改善されていく仕組み」** です。

### 仕組みの概要

2つのシンプルなルールで実現しています。

**ルール1：問題が起きたら `known_issues.md` に記録させる**

スキル定義のトラブルシューティングセクションに、以下を明記しています。

```
エラーや想定外の動作が発生した場合：
1. 原因を分析する
2. 対策を検討する
3. references/known_issues.md に記録する
```

**ルール2：起動時に必ず `known_issues.md` を読み込ませる**

ワークフローの Step 1 で、最初に `known_issues.md` を読み込むよう指示しています。

```
⓪ 過去の知見の確認
references/known_issues.md を read_file で読み込み、
過去に発生した問題と対策を把握する。
```

### なぜこれが効くのか

この2つのルールにより、以下のループが自律的に回ります。

> **問題発生 → 記録 → 次回起動時に読み込み → 同じ問題を回避**

セッションをまたいでも知見が引き継がれる。**「学習ループが自律的に回り、同じことで二度つまずかない設計」** をプロンプトレベルで実現しています。

## 😊 使ってみての実感

実際に使ってみると、**「サブエージェントを作成したい」と言うだけで対話形式で作成を完結できる** 体験は想像以上に気持ちが良いです。

そして、このアプローチの最大のメリットは **「改善ループの回しやすさ」** です。

作成支援をしてもらう中で「ここはもっとこう動いてほしい」という不満が出てきたら、`subagent-creator` のスキル定義（`SKILL.md` のプロンプトや制約）自体を修正します。

すると、**修正した効果は次のセッションですぐに有効になる**。

「改善ループを回せば、その恩恵を自分自身がすぐに受けられる」という体験が、さらにスキルを磨こうとする強力なインセンティブになっています。自分で使うツールを自分で改善するから、モチベーションが続くんです。


## 🎯 普遍的な学び：「スキル化」という考え方

`subagent-creator` を作って気づいたのは、これが単なる「便利ツールの紹介」ではないということです。

> **「何かやりたいことがあり、それを再現性高く繰り返したい。そして『やるたびに動作が改善されていく』ようにしたいなら、AI の Skill を作成して『学習ループが回る仕組み』を構築すると非常に効果的。」**

この考え方を図式化すると、こんなイメージです。

```
やりたいこと
  ↓
Skill を定義（= AI に仕事を覚えさせる）
  ↓
使う → 問題 / 改善点が見つかる
  ↓
Skill を修正
  ↓
次回から改善された動作が得られる（= 実施者にすぐ恩恵が返ってくる）
  ↓
さらに改善したくなる（= インセンティブが生まれる）
  ↓
...（ループ）
```

`subagent-creator` に限らず、**コードレビュー・テスト作成・ドキュメント生成・リリース確認**など、「毎回同じ手順でやるタスク」はすべてこの考え方で Skill 化できます。


## 📝 まとめ

- **Gemini CLI のサブエージェント**は、YAML フロントマター + Markdown で定義できる専門エージェント。experimental だが実用的。
- **「AI を作るための AI（Skill）」** を定義することで、毎回の作業を対話形式で効率化できる。
- **自律改善ループ**は「問題を記録する → 起動時に読み込む」2つのルールだけで実現できる。
- 「使うたびに恩恵が返ってくる」体験そのものが、改善を続けるインセンティブになる。
- **「スキル化」という考え方**は、あらゆる繰り返しタスクの自動化に応用可能。

気になった方は、ぜひ自分のプロジェクトで試してみてください。最初の一歩は、「自分が毎回やっているタスクは何か？」を考えるところから始まります 🚀
