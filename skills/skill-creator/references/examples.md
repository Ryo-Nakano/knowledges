# スキル完成例

スキルの `SKILL.md` を書くときの参考にする。  
`description` の書き方・ワークフローの構造・制約の記述の仕方を確認すること。

---

## 例 1: シンプル型 — コミットメッセージ生成スキル

```markdown
---
name: commit-message-writer
description: Git コミットメッセージの作成を支援する。ユーザーが「コミットメッセージを書いて」「コミット文を作って」「コミットどう書けばいい」のような意図を示したときに使用する。
---

# Commit Message Writer

変更内容を分析し、Conventional Commits 規約に準拠した Git コミットメッセージを提案する。

## ワークフロー

1. `list_directory` と `read_file` で変更ファイルを確認する
2. 変更の目的・スコープ・影響を把握する
3. 以下のフォーマットでコミットメッセージを提案する：

   ```
   <type>(<scope>): <subject>

   <body（任意）>
   ```

   - type: `feat` / `fix` / `docs` / `refactor` / `test` / `chore`
   - subject: 命令形・英語・50文字以内（日本語プロジェクトは日本語も可）

4. ユーザーが確認・修正し、最終的なメッセージを決定する
```

**ポイント：**
- シンプルな処理は 3〜4 ステップで完結させる
- `description` にユーザーが実際に使いそうな言葉を複数含める

## 例 2: ガイド型 — 仕様書作成スキル

```markdown
---
name: spec-writer
description: 機能の仕様書（スペック）を作成する。ユーザーが「機能の仕様を決めたい」「仕様書を書いて」「実装前に要件を整理したい」のような意図を示したときに使用する。
---

# Spec Writer

新機能や改善の仕様書をユーザーとの対話を通じて作成する。  
推測で進めず、不明な点は必ず確認すること。

## ワークフロー

### Phase 1: 目的と背景の確認

- 何を実現したいのか（ユーザーゴール）
- なぜ必要なのか（背景・課題）
- 誰が使うのか（対象ユーザー）

### Phase 2: 要件の具体化

- 機能の境界（何をするか・何をしないか）
- 正常系・異常系の動作
- データ・API・インターフェースの制約

### Phase 3: 仕様書ドラフトの作成

収集した情報をもとに仕様書のドラフトを作成し、ユーザーに確認する。

### Phase 4: ファイル出力

承認後、`write_file` で `specs/<feature-name>_spec.md` に保存する。

## 制約

- 推測・仮定で仕様を埋めない。不明な点は必ず質問する
- `specs/` ディレクトリ外への書き込みは行わない
```

**ポイント：**
- フェーズを分けることで長い作業も整理しやすい
- 制約セクションで安全な動作を保証する

## 例 3: インタラクティブ型 — 環境セットアップスキル

```markdown
---
name: env-setup-guide
description: 開発環境のセットアップを対話形式でガイドする。ユーザーが「環境を作って」「セットアップしたい」「開発準備をしたい」のような意図を示したときに使用する。
---

# Environment Setup Guide

開発環境のセットアップを、ユーザーの環境・目的に合わせてステップバイステップでガイドする。各ステップでユーザーの確認を取りながら進める。

## ワークフロー

### Phase 1: 環境の確認

現在の環境を確認する：
- OS・シェル・インストール済みのツールを `list_directory` / `read_file` で確認
- ユーザーに「何を開発したいか」「使いたい言語・フレームワーク」を聞く

### Phase 2: セットアップ手順の提案

環境情報をもとに必要なステップを提案する：
「現在の環境では〜をインストールする必要があります。以下の手順でいいですか？」

### Phase 3: 段階的なセットアップ

各ステップを1つずつ提示し、ユーザーの確認を得てから次に進む。  
エラーが発生した場合は原因を分析して代替手順を提案する。

### Phase 4: 完了確認

セットアップ完了後、動作確認コマンドを案内する。

## 制約

- ユーザーの確認なしにコマンドを連続実行しない
- `rm -rf`・`sudo` を含む破壊的なコマンドは提案前に必ず警告する
- ホームディレクトリ以外への書き込みは行わない
```

**ポイント：**
- 確認を随所に挟む設計で安全性を確保
- エラー対応をフローに組み込む
