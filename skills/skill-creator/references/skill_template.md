# SKILL.md テンプレート

スキル生成時はこのテンプレートをベースに、ヒアリング内容で各プレースホルダーを埋める。

---

## テンプレート本体

```markdown
---
name: [小文字・ハイフン区切りのスキル識別子]
description: [三人称・具体的キーワード入り。「〜を支援する。ユーザーが〜と言ったときに使用する。」の形で書く]
---

# [スキルの表示名]

[このスキルの役割を1〜2文で説明する。エージェントがこのブロックを読んで何をすべきか理解できること。]

> **重要：ユーザーとのすべてのやり取りは日本語で行うこと。**（必要に応じて削除可）

---

## ワークフロー

### Phase 1: [最初のフェーズ名]

[このフェーズでやること。箇条書きで明確に。]

### Phase 2: [次のフェーズ名]

[このフェーズでやること。]

（フェーズ数はスキルの複雑さに応じて増減する）

---

## 注意事項・制約

- [守るべきルールや禁止事項を箇条書きで記述]
- [ファイル操作が発生する場合は、書き込み先の制限を明記する]
```

---

## 各フィールドの書き方ガイド

### `name`
- 小文字とハイフンのみ（例: `code-reviewer`, `spec-writer`）
- スペース・大文字・記号は使わない
- 目的を端的に表す名詞+動詞の構成が読みやすい

### `description`
- Antigravity がルーティング判断に使う**最重要フィールド**
- 以下の構成で書く：
  ```
  [スキルの機能を三人称で説明]。[どんなユーザー発言で起動するか、具体的なキーワードを列挙]したときに使用する。
  ```
- 良い例:
  ```
  コードの品質・可読性・バグの可能性をレビューする。ユーザーが「コードを見てほしい」「レビューして」「品質チェックして」のような意図を示したときに使用する。
  ```
- 悪い例（曖昧すぎる）:
  ```
  コードを確認する。コードに関することがあれば使う。
  ```

### 本文（システムプロンプト）
- **役割**: エージェントに何者なのか・何をするのかを最初に伝える
- **ワークフロー**: フェーズに分けて手順を明確に記述する
- **制約**: やってはいけないこと・守るべきルールを明記する
- **出力フォーマット**: 期待する出力形式がある場合は具体的に書く

---

## ワークフロー選択ガイド

| パターン | いつ使うか | 例 |
|---|---|---|
| **シンプル型**（1〜3ステップ） | 繰り返す単純な処理 | フォーマット変換・テキスト整形 |
| **ガイド型**（フェーズ分け） | 意思決定が必要なタスク | 設計・レビュー・仕様作成 |
| **インタラクティブ型**（確認挟み） | 安全性重視・ユーザー判断が必要 | ファイル生成・コード変更 |
