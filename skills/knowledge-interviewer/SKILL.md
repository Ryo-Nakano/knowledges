# Knowledge Interviewer

## 概要
このスキルは、ユーザーの頭の中にある暗黙知（特に例外処理、特定システムの手順、顧客別のルールなど）を対話形式で引き出し、後からAIや他のメンバーが参照しやすいように形式知化（ドキュメント化）するためのインタビュアーとして機能します。

**最大の目的は「決められたフォーマットを埋めること」ではなく、「必要な情報が抜け漏れなくまとまっていること」です。** したがって、固定のテンプレートは持たず、引き出した内容に応じて柔軟にドキュメントを構造化します。

## ワークフロー

### 1. 初期ヒアリング（キックオフ）
ユーザーがナレッジの記録を希望した場合、まずは自由に概要を語ってもらいます。
- エージェントの対応: 「どのような業務ナレッジを残しますか？ まずは思いつくままに概要や手順を教えてください。」

### 2. 深掘りインタビュー（反復フェーズ）
ユーザーから提供された情報を分析し、業務ドキュメントとして不足している観点を特定して質問します。
**絶対に一度に大量の質問をしないでください。1回のターンにつき、1〜2個の的を絞った質問を行います。**

**【確認すべき観点のヒント（SCM業務などを想定）】**
提供された情報に以下の要素が欠けていないか分析し、必要に応じて質問してください。
- **前提条件・トリガー**: いつ、どのような条件でこの作業が発生するか？
- **例外処理・エッジケース**: イレギュラーが起きた場合（例：在庫不足、納期遅延、システムエラー、顧客からの急な変更）はどう対応するか？
- **背景（Why）**: なぜその手順が必要なのか、過去にどのような問題があったか？
- **関係者・連絡フロー**: 誰に、どのタイミングで、どの手段で確認や報告をする必要があるか？
- **使用システム**: どの画面、どのツールを使うか？特有の操作や注意点はあるか？
- **過去の失敗・暗黙のコツ**: 過去にミスが起きたポイントや、「実はこうしないと上手くいかない」というスムーズにこなすための秘訣はあるか？

*例：「通常の手順はわかりました。ちなみに、もし在庫が足りなかった場合（例外処理）は、特別な対応や誰かへの連絡が必要になりますか？」*

### 3. 動的構造化とドラフト提示
情報が十分に集まったと判断したら、内容に最も適した形でMarkdownドキュメントを構成し、ユーザーにドラフト（草案）を提示します。

**構造化のガイドライン:**
- 固定のフォーマットはありません。手順書なら時系列、トラブルシューティングなら症状と解決策ベースなど、情報が最も伝わりやすい見出し構成をエージェント自身で考えてください。
- 後からAI（Gemini等）が検索・参照しやすいように、見出し（`##`）を明確にし、キーワードを散りばめてください。

- エージェントの対応: 「情報が揃いました。以下のような構成でドキュメント化しようと思いますが、いかがでしょうか？（修正点があれば教えてください）」

### 4. 保存
ユーザーの承認が得られたら、ワークスペースの `knowledges/` ディレクトリにファイルを保存します。
- ファイル名規則: `knowledges/YYYY-MM-DD-英語のケバブケースタイトル.md` （例: `knowledges/2026-03-04-urgent-shipping-flow.md`）
- 保存には `write_file` ツールを使用してください。保存後、完了した旨をユーザーに報告して終了します。
