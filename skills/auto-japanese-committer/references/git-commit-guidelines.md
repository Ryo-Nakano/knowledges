# Git コミットの粒度とメッセージのガイドライン

本ドキュメントでは、チーム開発においてコードの品質と保守性を高めるための、Gitコミットの粒度とコミットメッセージのベストプラクティスをまとめます。
特に「**コードレビューの容易性**」と「**チケット駆動開発**」を意識した運用を前提としています。

---

## 1. コミット粒度の基準

コミットの粒度（大きさ）は、**レビュアーが「変更の意図」を理解しやすく、後から変更を追いやすい（または取り消しやすい）単位**であることが求められます。

### レビュー容易性の重視
- **レビュアーの認知的負荷を下げる**
  一度に大量の変更（数百行以上の差分や複数ファイルにまたがる全く別の修正）が1つのコミットにまとまっていると、レビューの難易度が跳ね上がります。大きな機能追加、バグ修正、デザインの調整などは、できるだけ別のコミットに分割してください。
- **「関心の分離」を意識する**
  例えば、「新しいAPIエンドポイントの作成」と「既存の不要なコードの削除」は、それぞれ別のコミットにするとレビューがしやすくなります。

### アトミックコミット（1コミット = 1つの論理的変更）の意識
- 1つのコミットに1つの論理的な変更のみを含める（アトミックコミット）のが理想的ですが、**厳密に徹底する必要はありません**。
- ただし、バグ修正と大規模なリファクタリングを同じコミットに混ぜるなど、後から見たときに意図が分かりにくくなるような混在は避けてください。
- 変更がある程度独立していれば、後から特定の変更だけを取り消す（`git revert`）ことや、バグの原因箇所を特定する（`git bisect`）ことが容易になります。
- 巨大な修正になってしまった場合は、`git add -p` などを活用して論理的な単位でステージングし、分割してコミットすることを推奨します。

### チケット駆動開発との連携
- コミットは**可能な限り**、JiraやGitHub Issueなどの「チケット（タスク）」に関連づけて行います。
- ただし、軽微なタイポの修正や、チケットを立てるほどではない小さな改善など、**必ずしもすべてのコミットがチケットに紐づく必要はありません**。
- 作業途中の「とりあえず保存」的なコミット（例: `wip`, `update`）は避け、もし行った場合は Pull Request をマージする前に `git rebase -i` などで squash（結合）し、履歴を整理してください。

## 2. コミットメッセージのベストプラクティス

コミットメッセージは、**Conventional Commits**（プレフィックスを用いた世界的標準フォーマット）を採用します。これにより、機械的な解析（Changelogの自動生成など）が可能になり、履歴の一覧性も向上します。

### 基本構造

```text
<type>(<scope>): <subject> (#Issue番号)

<body>

<footer>
```

*※ `scope`, `body`, `footer` は省略可能ですが、重要な変更にはできるだけ `body` を記述します。*

### Type（プレフィックス）の種類
変更の目的に応じて、以下のプレフィックスを使い分けます。

| Type | 説明 |
| :--- | :--- |
| **`feat`** | 新機能の追加 |
| **`fix`** | バグ修正 |
| **`docs`** | ドキュメントのみの追加・修正（READMEなど） |
| **`style`** | コードの動作に影響を与えないフォーマット修正（スペース、セミコロンなど） |
| **`refactor`** | バグ修正も機能追加も伴わない、コードの構造変更（リファクタリング） |
| **`perf`** | パフォーマンスを向上させる変更 |
| **`test`** | テストコードの追加・修正 |
| **`chore`** | プロダクションコードに影響しない、ビルドプロセスやライブラリ更新など |

### Subject（タイトル・1行目）のルール
- **50文字以内に収める:** GitHub等のUIで省略されずに表示させるため。
- **チケット番号を含める（任意）:** チケットが存在するタスクに対する変更の場合は、どのタスクに対する変更か即座にわかるように末尾などに `#Issue番号` を付与します（例: `feat: ログイン画面のUI実装 (#123)`）。
- **簡潔に「何をしたか」を書く:** 読んだだけで影響範囲と目的が伝わるようにする。

### Body（本文）のルール
- **「なぜ（Why）」この変更をしたかを書く:** 「何をしたか（What）」は `git diff` を見れば分かります。しかし、「なぜ既存の実装を変えたのか」「なぜこのアルゴリズムを選択したのか」といった背景はメッセージにしか残せません。
- **72文字程度で改行する:** ターミナルで `git log` を見た際の視認性を保つため。
- タイトル行とは必ず1行の空行を空けて書き始めます。

### Footer（フッター）のルール
- チケットをクローズする場合は `Closes #123` や `Fixes #123` を記載します。
- 後方互換性を壊す変更（破壊的変更）が含まれる場合は `BREAKING CHANGE:` と明記し、移行手順などを記載します。
